An Introduction to Three Tiered Applications

Please review the concepts covered this module.

Three Tiered Applications

Three tiered applications consist of three main parts that work together to accomplish the same goal: to create a robust application that stores, manipulates, and retrieves data for a user to interact with. They are incredibly popular on the web. Not all applications are three tiered, and some have more than three tiers.

The three tiers are:

  • The client side tier (also known as the presentation or user interface tier)
  • The server tier (also known as the business logic tier)
  • The data storage tier (also known as the database tier)

The three tiers work together to create, read, update, and delete data, as well as display data to the end user.

Let's break the three tiers down a bit further.

The Client-side Tier

The client side tier is anything that an end user sees or interacts with and is usually known as the user interface. It is usually written using HTML, CSS, and JavaScript, or using frameworks such as TailwindCSS or React. An example that has been built using React is the front page of Facebook. When you "like" a post on Facebook, the client-side tier notifies the server tier of the associated change.

The Server Tier

The server tier is where the business logic resides within an application. This section is mainly concerned with routing information from the client to the database. It also makes sure the data is processed correctly. For example, when you "like" a post on facebook, the server tier is responsible for making the appropriate change in the database.

Data Storage Tier

The database is where data is stored. The server retrieves data from the database and manipulates it according to the messages it receives from the client-side tier. It is important to remember that data is merely stored in a database. It is not manipulated in any way, that happens on the server tier. An example of a database would be a data center which Facebook uses to store user data such as a their name, email, and, in this case, history of what they have "liked".

Unit Testing

At first, unit testing can seem overwhelming. You start by identifying a small piece of functionality to test. Use the setup, expected output, and assertion methods to write your test.

Here's an example that tests if a function, getUserName() returns an expected output of "Paul".

test("username can be retrieved", () => {
  const user = new User("Paul"); 
  const userName = getUserName(user)

  expect(userName).toBe("Paul"); 
});

First, you setup the test: const user = new User("Paul"). This sets the expectation that a new user is created to use in the test.

You then establish the expected result: const userName = getUserName(user). This sets the expectation that the getUserName function will equal the variable, userName, in this case, the stored string of "Paul".

Finally, you create the assertion. This will check that what you expect to happen is what you were expecting to happen. Use Jest's assertions, such as .toBe or .toEqual to write code that checks if the returned output you expect is equal to the output that is returned. If they match, the test has shown the code works. However, if they fail to match, you need to return to your code and fix it appropriately.
